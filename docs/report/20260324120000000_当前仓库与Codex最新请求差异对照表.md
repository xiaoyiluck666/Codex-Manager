# 当前仓库与 Codex 最新请求差异对照表

说明：这里只列“真正影响出站请求语义”的差异，像 `Authorization`、`ChatGPT-Account-ID`、`x-openai-subagent`、`x-codex-beta-features`、`x-codex-turn-metadata`、`x-openai-internal-codex-residency`、请求压缩这类两边已基本对齐的项不重复列出。

| 差异项 | Codex 最新源码 | 当前仓库 | 影响 |
| --- | --- | --- | --- |
| `User-Agent` | `codex_cli_rs/<env!("CARGO_PKG_VERSION")> (...)`，版本号来自编译时包版本 | `codex_cli_rs/<数据库/运行时配置版本> (...)`，版本号可运行时调整 | 当前仓库可以手动追平，但来源不是编译时版本，容易和上游不一致。 |
| `originator` | 由 `codex_login::default_client::originator()` 统一生成，支持环境变量/初始化阶段覆盖 | 出站固定走 `current_wire_originator()`，本地 `gatewayOriginator` 只保留在配置层 | 当前仓库的“可配置 originator”不会真正影响出站请求。 |
| `x-client-request-id` | 始终等于 `conversation_id` | 已改为优先保留原始 `conversation_id`；无 `conversation_id` 时才退回既有值 | 外发请求不再被线程锚点直接污染，和官方更接近。 |
| `session_id` | 始终等于 `conversation_id` | 已改为优先保留原始 `conversation_id`；无 `conversation_id` 时才退回既有值 | 会话标识已回到以本轮 `conversation_id` 为主。 |
| `x-codex-turn-state` | 同一 turn 内持续回放 | 仅在稳定线程锚点存在且与 `conversation_id` 不冲突时保留；冲突、切线程或孤儿状态会被清掉 | 当前仓库仍然更保守，避免跨线程粘连。 |
| `prompt_cache_key`（body） | Responses 请求里通常固定为 `conversation_id` | 会被网关根据线程锚点/强制键重写或补写 | 当前仓库的 thread 语义仍偏“网关驱动”，不是官方直连语义。 |
| 请求路径形态 | 代码层直接走 `responses` / `responses/compact`，由 base URL 拼出最终地址 | 外层暴露 `/v1/responses`，再由网关重写到 `.../backend-api/codex/...` | 最终上游地址基本一致，但当前仓库多了一层路径改写。 |

## 结论

1. 目前最核心的差异已收敛到“回放/线程锚点”处理：`x-codex-turn-state`、`prompt_cache_key` 这两项仍被网关逻辑重新解释。
2. `User-Agent` 的版本来源不同，是另一个稳定差异点。
3. 其余常见请求头基本已经和 Codex 官方实现对齐。
