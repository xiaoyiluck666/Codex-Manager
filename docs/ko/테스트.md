#테스트 중

이 문서는 CodexManager 저장소 수준의 테스트 및 검증 기준을 정의합니다.

목표:

- 새로운 공동 작업자는 적어도 변경 후에는 무엇을 실행할지 빠르게 알 수 있습니다.
- 모든 변경 사항을 전체 검증으로 업그레이드하지 마세요.
- 프로토콜 호환성, 링크 게시, 설정 관리 등 위험도가 높은 변경에 대해 고정된 검사 입구 제공

## 1. 기본 환경

-Node.js 20
- pnpm 9
- 녹이 안정됨
- PowerShell 7+(Windows 패키징/스크립트 유효성 검사)

## 2. 프런트엔드 변경 사항

적용 범위:

- `apps/src/app/`
- `apps/src/components/`
- `apps/src/lib/`
- `apps/src/hooks/`

최소한의 검증:

``배쉬
pnpm -C 앱 실행 빌드
pnpm -C 앱 실행 테스트:런타임
````

설명:

- `pnpm -C apps run build`: Next.js 정적 내보내기 링크가 여전히 정상인지 확인
- `pnpm -C apps run test:runtime`: 런타임 성능 판단 및 데스크탑 확인 / Web 성능 저하 로직이 반환되지 않음
- 변경 사항에 런타임 식별, Web RPC, 데스크톱/Web 차이 처리가 포함된 경우 섹션 4를 구성합니다.

## 3. 데스크톱 / Tauri 변경 사항

적용 범위:

- `apps/src-tauri/`
- 데스크탑 업데이트, 트레이, 창, 커맨드 브릿지 관련 변경 사항

최소한의 검증:

``배쉬
화물 테스트 --workspace
````

추가 제안:

``파워셸
pwsh -NoLogo -NoProfile -파일 스크립트/rebuild.ps1 -Bundle nsis -CleanDist
````

설명:

- Tauri 브리징 또는 데스크톱 수명주기를 변경할 때마다 로컬 데스크톱 빌드 확인을 한 번 이상 수행하는 것이 좋습니다.

## 4. Web 셸/배포 호환 변경 사항 실행

적용 범위:

- `crates/web/`
- `apps/src/lib/api/transport.ts`
- `apps/src/components/layout/app-bootstrap.tsx`
- `apps/src/components/layout/header.tsx`
- `apps/src/components/layout/sidebar.tsx`
- Web 에이전트, `/api/runtime`, `/api/rpc` 및 배포 방법 관련 변경 사항

최소한의 검증:

``배쉬
pnpm -C 앱 실행 빌드
pnpm -C 앱 실행 테스트:런타임
화물 테스트 -p codexmanager-web
pwsh -NoLogo -NoProfile -파일 스크립트/tests/web_runtime_probe.test.ps1
````

제안된 추가 사항:

``파워셸
pwsh -NoLogo -NoProfile -파일 스크립트/tests/web_runtime_probe.ps1 `
  -기지 http://localhost:48761
pwsh -NoLogo -NoProfile -파일 스크립트/tests/web_ui_smoke.ps1 -SkipBuild
````

설명:

- `pnpm -C apps run build`: 프런트엔드 정적 내보내기가 여전히 생성될 수 있는지 확인
- `pnpm -C apps run test:runtime`: 프런트엔드 런타임 계약과 기능 결정이 일치하는지 확인
- `cargo test -p codexmanager-web`: Web 쉘 라우팅 및 런타임 프로브 계약 확인
- `web_runtime_probe.test.ps1`: 쉘 최소 스모크 링크 실행을 위한 Web 스크립트 동작 확인
- `web_ui_smoke.ps1`: 지원/지원되지 않는 실행 셸에서 Web 페이지의 주요 UI 동작을 확인합니다.
````

````
## 6. 프로토콜 적응/게이트웨이 수정

적용 범위:

- `crates/service/src/gateway/`
- `crates/service/src/http/`
- `crates/service/src/lib.rs`

반드시 다루어야 할 사항:

- `/v1/responses`
- `/v1/chat/completions`
- 스트리밍 SSE
- 비스트리밍 JSON
- `tools`
- `tool_calls`

최소한의 검증:

``배쉬
화물 테스트 --workspace
pwsh -NoLogo -NoProfile -파일 스크립트/tests/gateway_regression_suite.ps1
pwsh -NoLogo -NoProfile -파일 스크립트/tests/codex_stream_probe.ps1
pwsh -NoLogo -NoProfile -파일 스크립트/tests/chat_tools_hit_probe.ps1
````

설명:

- 로컬 환경에 실제 업스트림 계정이 없다면 최소한 Rust 테스트를 실행하고 프로브 실행 지침을 유지하세요.
- 한 종류의 클라이언트만 검증할 수 없는 호환성 수정.

## 7. 설정/환경 변수/지속성 변경

적용 범위:

- `apps/src/settings/`
- `crates/service/src/app_settings/`
- `crates/core/src/storage/settings.rs`
- `CODEXMANAGER_*` 구성 항목 추가

최소한의 검증:

``배쉬
pnpm -C 앱 실행 빌드
화물 테스트 --workspace
````

수동으로 확인해야 합니다.

- 기본값이 명확한가?
- `app_settings` 작성 여부
- 런타임에 동기화가 필요한지 여부
- README / `기여-가이드.md` / `아키텍처.md` 업데이트가 필요합니까?

``배쉬
````

## 11. 결과 기록 규칙

- 완전히 실행 가능한 검증은 "실행됨"으로 기록됩니다.
- 환경 제한으로 인해 실행할 수 없는 검증은 "실행되지 않음 + 이유"로 명확하게 표시됩니다.
- '통과할 수 있어야 한다'를 '검증됐다'로 착각하지 마세요.
