# FAQ 및 계정 적중 규칙

## FAQ
- 승인 콜백 실패: `CODEXMANAGER_LOGIN_ADDR` 점유 여부를 우선적으로 확인하거나 UI에서 수동 콜백 해결을 사용합니다.
- 모델 목록 또는 요청이 챌린지에 의해 차단됩니다. 프록시 종료, 헤더 차이 및 계정 상태 요청을 우선적으로 확인하고 더 이상 이전 대체 또는 업스트림 쿠키를 통해 문제를 해결하지 않습니다.
- 여전히 Cloudflare/WAF에 의해 차단됨: 이전 호환성 경로를 따라 문제를 해결하지 말고 앞으로는 `Codex-First`의 의미에 따라서만 진행하세요.
- "일부 데이터 새로 고침에 실패했습니다. 사용 가능한 데이터가 표시되었습니다."라는 메시지가 자주 나타납니다. 자동 새로 고침 시나리오가 기록 로그만 표시되도록 변경되었습니다. 수동으로 새로 고치면 실패한 항목과 샘플 오류가 표시됩니다. 설정 페이지에서 "백그라운드 작업" 간격/스위치를 확인하고, 서비스 로그에서 실패한 작업 이름을 우선적으로 확인하세요.
- 독립적으로 서비스/Web 실행: 디렉터리(예: 설치 디렉터리)를 쓸 수 없는 경우 `CODEXMANAGER_DB_PATH`를 쓰기 가능한 경로로 설정하세요.
- macOS 프록시 환경에서의 요청 `502/503`: 시스템 프록시가 로컬 루프백 요청을 인계받지 않는지 확인하고(`localhost/127.0.0.1` `DIRECT`로 이동) 주소가 소문자 `localhost:<port>`을 사용하는지 확인합니다.

## 마이그레이션 지침

### Codex-첫 번째 방향

현재 저장소는 후속 방향을 `Codex-First`로 확정했습니다.

- 더 이상 기존의 호환 동작을 메인라인 경로로 사용하지 않습니다.
- 하나의 활성 계정만 동일한 세션에 바인딩될 수 있습니다.
- 수동으로 계정을 전환하면 스레드가 잘리고, 자동으로 계정을 전환하면 스레드도 잘립니다.

### 현재 행동과 목표 행동의 차이

현재 동작:

- `balanced` `Key + 模型`에 따라 여전히 엄격한 폴링
- 세션 및 계정의 바인딩 의미는 여전히 "요청별 폴링"에서 "세션별 바인딩"으로 마이그레이션되고 있습니다.

목표 행동:

- 바운드 세션은 먼저 바운드 계정에 도달하며 더 이상 각 일반 폴링 라운드에 참여하지 않습니다.
- 자동으로 계정을 전환하면 업스트림 스레드 생성이 동기적으로 전환됩니다.
- 기존 호환 스위치는 기본 경로에서 완전히 빠져나와 더 이상 권장 구성으로 사용되지 않습니다.

## 계정 적중 규칙
- `ordered`(순서 우선 순위) 모드에서 게이트웨이는 계정 `sort`의 오름차순으로 후보를 만들고 순서대로 시도합니다(예: `0 -> 1 -> 2 -> 3`).
- 이는 "항상 숫자 0을 누르는 것"이 ​​아니라 "순서대로 시도"하는 것을 의미합니다. 이전 계정을 사용할 수 없거나 실패하면 자동으로 다음 계정으로 전환됩니다.

### 선주문 계정이 히트하지 못하는 일반적인 이유
- 계정 상태가 `active`이 아닙니다.
- 계정에 토큰이 없습니다.
- 메인창이 소진되어 용도 항목이 누락되는 등 용도 판단이 불가능합니다.
- 계정이 휴지 상태이거나 동시 소프트 캡으로 인해 건너뛰기가 트리거됩니다.

### `balanced` 모드
- `balanced`(균형 라운드 로빈) 모드는 기본적으로 최소 `sort`에서 시작한다는 보장 없이 `Key + 模型` 차원에 따라 사용 가능한 모든 계정 간에 엄격하게 순환됩니다.
- 체력 교환은 `CODEXMANAGER_ROUTE_HEALTH_P2C_BALANCED_WINDOW`을 명시적으로 늘리는 경우에만 밸런스 폴 헤드에 추가됩니다.

## 문제 해결 로그
데이터베이스 `gateway-trace.log`의 동일한 디렉터리를 볼 수 있습니다.
- `CANDIDATE_POOL`: 이 요청에 대한 후보자 명령
- `CANDIDATE_START` / `CANDIDATE_SKIP`: 실제 시도와 건너뛰기 이유
- `REQUEST_FINAL`: 최종 히트 계정
