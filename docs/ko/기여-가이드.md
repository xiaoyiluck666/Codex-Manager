# 기여

이 문서는 새로운 협력자가 구두 전달을 최소화하면서 개발, 확인, 제출 및 릴리스를 완료할 수 있도록 하는 것을 목표로 CodexManager의 일일 협력 방법을 제한하는 데 사용됩니다.

## 1. 프로젝트 포지셔닝

CodexManager 단일 프런트 엔드 프로젝트도 아니고 단일 Rust 서비스 프로젝트도 아닙니다.
현재 저장소에는 다음 문서도 포함되어 있습니다.

- 데스크톱: `apps/` + `apps/src-tauri/`
- 지역 서비스: `crates/service`
- Web 쉘: `crates/web`
- Service 런처: `crates/start`
- 데이터 및 저장 기반 : `crates/core`
- 빌드/릴리스 스크립트: `scripts/`
- GitHub 액션 게시 링크: `.github/workflows/`

따라서 제출하기 전에 먼저 변경 사항이 속하는 경계를 결정하고 여러 책임을 동일한 파일에 직접 쌓지 않도록 해야 합니다.

거버넌스 문서 항목:

- `README.md`: 프로젝트 소개 및 빠른 시작
- `아키텍처.md`: 구조적 경계 및 운영 관계
- `테스트.md`: 저장소 수준 검증 기준
- `보안.md`: 민감한 정보 처리에 관한 보안 문제 및 규칙
- `docs/README.md`: 거버넌스 문서 디렉토리 및 제출 규칙

## 2. 개발 환경

### 2.1 필수 도구

-Node.js 20
- pnpm 9
- 녹이 안정됨
- Windows 로컬 패키징에는 PowerShell 7 이상이 필요합니다.
- Tauri 패키징은 플랫폼 종속성과 일치해야 합니다.

### 2.2 종속성 설치

``배쉬
pnpm -C 앱 설치
cargo test --workspace
````

### 2.3 일반적으로 사용되는 로컬 명령어

프런트엔드:

``배쉬
pnpm -C 앱 개발 실행
pnpm -C 앱 실행 테스트
pnpm -C 앱 실행 test:ui
pnpm -C 앱 실행 빌드
pnpm -C 앱 실행 확인
````

녹:

``배쉬
화물 테스트 --workspace
화물 빌드 -p codexmanager-service --release
화물 빌드 -p codexmanager-web --release
화물 빌드 -p codexmanager-start --release
````

데스크탑 포장:

``파워셸
pwsh -NoLogo -NoProfile -파일 스크립트/rebuild.ps1 -Bundle nsis -CleanDist -Portable
````

## 3. 커밋 경계

### 3.1 책임에 따라 파일 수정

다음 경계의 우선순위를 지정하십시오.

- 프런트엔드 페이지, 상호작용, 상태: `apps/src/`
- 데스크탑 셸, 트레이, 창, Tauri 명령: `apps/src-tauri/src/`
- 서버 측 HTTP / RPC / 게이트웨이 / 프로토콜 적응: `crates/service/src/`
- 데이터베이스 마이그레이션, 스토리지 인프라: `crates/core/`
- 릴리스 및 빌드 스크립트: `scripts/`, `.github/workflows/`

### 3.2 현재 고위험 파일

다음 파일은 확실히 너무 크므로 수정할 때 일반 제어 논리를 추가하지 않아야 합니다.

- `apps/src/main.js`
- `apps/src-tauri/src/lib.rs`
- `crates/service/src/lib.rs`
- `crates/service/src/gateway/protocol_adapter/response_conversion.rs`
- `.github/workflows/release-all.yml`

### 3.3 대용량 파일 경고 임계값

When the following threshold is reached, the logic should not be continued to be piled in by default, but the split should be evaluated first:

- JavaScript/TypeScript: 경고는 `500`를 초과하는 줄에서 시작되며 `800`을 초과하는 줄에서는 분할되지 않는 이유를 설명해야 합니다.
- Rust: `400`을 초과하는 줄에 대해 경고가 시작됩니다. `700`을 초과하는 노선은 철거하지 말아야 하는 이유를 설명해야 합니다.
- 워크플로/YAML: `250`을 초과하는 줄에 대해 경고가 시작됩니다. `400`을 초과하는 줄은 분할되지 않는 이유를 설명해야 합니다.
- 마크다운 문서: 줄이 `300`을 초과하면 경고가 시작되고 `docs/` 하위 문서에 우선순위가 부여됩니다.

설명:

- '경고 시작'은 제출하기 전에 책임 분할을 계속할지 여부를 적극적으로 결정해야 함을 의미합니다.
- "분할하지 않는 이유를 설명해야 한다"는 것은 제출 지침이나 PR 설명에 그 이유를 명확하게 기재해야 함을 의미합니다.
- 이러한 임계값은 일회성 정리 지표가 아닌 장기적인 유지 관리 제약 조건입니다.

### 3.4 금지 품목

- 메인 입구에 설정, 이벤트 바인딩, 프로토콜 브랜치를 계속 쌓아두지 마세요.
- README를 장기적인 유지 관리를 위한 변경 로그로 취급하지 마십시오.
- 확인 없이 스크립트, 작업 흐름 또는 버전 번호를 변경하지 마십시오.
- 생성하지 않은 사용자 변경 사항을 롤백하지 마십시오.
- 릴리스 워크플로우에서 인라인 스크립트를 다시 복사 및 확장하지 말고 먼저 재사용하십시오(`scripts/release/`).

## 4. 제출 전 확인

### 4.1 최소 체크리스트

변경 범위에 따라 최소한 다음 내용을 수행하십시오.

프런트엔드 변경 사항:

``배쉬
pnpm -C 앱 실행 테스트
pnpm -C 앱 실행 빌드
pnpm -C 앱 실행 test:ui
````

Rust/서버측 변경 사항:

``배쉬
화물 테스트 --workspace
````

데스크탑/패키징 링크 변경:

``파워셸
pwsh -NoLogo -NoProfile -파일 스크립트/rebuild.ps1 -DryRun
````

### 4.2 프로토콜 적응과 관련된 변경 사항

다음 경로가 변경되면 최소한의 회귀 검증을 수행해야 합니다.

- `crates/service/src/gateway/`
- `crates/service/src/http/`
- `crates/service/src/lib.rs`

최소 적용 범위:

- `/v1/chat/completions`
- `/v1/responses`
- 스트리밍 수익
- 비 스트리밍 반환
- `tool_calls`/도구 관련 경로

### 4.3 설정 항목 관련 변경 사항

설정 페이지 필드, 환경 변수 또는 지속성 구성을 추가하는 경우 다음 사항도 확인해야 합니다.

- 기본값이 명확한가?
- `app_settings` 작성이 필요한지 여부
- 데스크톱/서비스/웹 단말의 동작에 영향을 미치는지 여부
- README 또는 전용 문서를 업데이트해야 하는지 여부

## 5. 정보 및 홍보 동의서 제출

### 5.1 정보 제출

현재 저장소는 주로 중국어 제출 지침을 기준으로 운영되며, 다음 사항을 요구합니다.

- 제출물은 한 가지 유형의 문제만 해결합니다.
- 제목은 결과를 직접적으로 설명하며 빈 단어를 쓰지 않습니다.
- 서로 관련되지 않은 여러 변경 사항을 동일한 커밋에 넣지 마세요.

### 5.2 PR은 최소 요구 사항을 설명합니다.

PR은 최소한 다음 사항을 명확하게 명시해야 합니다.

- 어떤 파일이 변경되었나요?
- 어떤 문제를 해결해야 할까?
- 영향을 받는 플랫폼이나 인터페이스
- 어떤 검증이 실행되었나요?
- 보장되지 않을 위험이 있나요?

## 6. 출시 전 점검

각 릴리스 전에 다음을 확인해야 합니다.

1. `변경-이력.md` 업데이트.
2. `README.md`은 `README.en.md`의 현재 버전 항목과 일치합니다.
3. `Cargo.toml`, `apps/src-tauri/Cargo.toml` 및 `apps/src-tauri/tauri.conf.json` 버전과 일치합니다.
4. 릴리스 워크플로 입력 설명, 스크립트 매개 변수 설명 및 실제 워크플로가 일관되어야 합니다.
5. 고위험 호환성 경로는 최소 1회 이상의 현지 검증을 완료해야 합니다.
6. 제품 이름 지정 또는 릴리스 유형 논리가 변경되면 `prerelease` 및 태그 동작을 확인해야 합니다.

## 7. 문서 관리 규칙

The long-term maintenance agreement is as follows:

- `README.md` / `README.en.md` 프로젝트 소개, 퀵 스타트, 참가 안내 등을 담당합니다.
- `변경-이력.md` 버전 기록을 담당합니다.
- `아키텍처.md` 구조적 경계 및 운영 관계를 담당합니다.
- `기여-가이드.md` 협업 규칙 및 제출 전 확인을 담당합니다.

버전 기록, 아키텍처 설명, 릴리스 세부 정보를 README에 다시 쌓지 마세요.

## 8. 주요 변화에 대처하는 방법

다음 조건 중 하나라도 충족되면 작업을 제출하기 전에 작업을 분할하는 것이 좋습니다.

- 프런트엔드, 데스크톱, 서버의 세 가지 경계를 동시에 포함
- 프로토콜 적응 변경, 지속성 설정 및 링크 게시를 동시에 수행
- 제품명 변경, 워크플로우 수정, 버전 전략 조정 필요
- 고위험 대용량 파일 분할 필요

권장 순서:

1. 먼저 테스트 또는 검증 스크립트를 보완합니다.
2. 재건축이나 구조조정을 다시 하세요
3. 최종 문서 및 버전 참고 사항
