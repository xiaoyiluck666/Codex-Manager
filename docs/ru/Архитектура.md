#АРХИТЕКТУРА

В этом документе описываются текущая структура репозитория CodexManager, связи между частями системы и релизный контур. Его цель — помочь участникам быстро понять, в какой слой проекта нужно вносить изменения.

## 1. Общая форма

CodexManager состоит из двух типов режимов работы:

1. Режим рабочего стола: Tauri Рабочий стол + локальный сервисный процесс
2. Режим Service: автономный сервис + веб-интерфейс, может использоваться с сервером, Docker или без среды рабочего стола.

Единая цель:

- Управление учетными записями, использованием и ключами платформы
- Обеспечение возможностей локального шлюза
- Внешне совместим с записью в стиле OpenAI и адаптирован к нескольким восходящим протоколам.

## 2. Структура и обязанности директории

```text
.
├─ apps/                  # 前端与 Tauri 桌面端
│  ├─ src/                # Vite + 原生 JavaScript 前端
│  ├─ src-tauri/          # Tauri 桌面壳与原生命令桥接
│  ├─ tests/              # 前端 UI/结构测试
│  └─ dist/               # 前端构建产物
├─ crates/
│  ├─ core/               # 数据库迁移、存储基础、认证/用量底层能力
│  ├─ service/            # 本地 HTTP/RPC 服务、网关、协议适配、设置持久化
│  ├─ web/                # Web UI 服务壳，可嵌入前端静态资源
│  └─ start/              # Service 一键启动器（拉起 service + web）
├─ scripts/               # 本地构建、统一版本、测试探针、发布辅助脚本
├─ docker/                # Dockerfile 与 compose 配置
├─ assets/                # README 图片、Logo 等静态资源
└─ .github/workflows/     # CI / release workflow
```

## 3. Базовый комплексный индекс записи домена

### 3.1 Главный вход внешнего управления

- `apps/src/main.js`: Вход в стартовую сборку внешнего интерфейса
- `apps/src/runtime/app-bootstrap.js`: Порядок инициализации интерфейса
- `apps/src/runtime/app-runtime.js`: Координация процесса обновления и времени выполнения.
- `apps/src/settings/controller.js`: Настройка фасада домена и продолжение распространения по подмодулям

### 3.2 Вход в оболочку рабочего стола

- `apps/src-tauri/src/lib.rs`: Tauri запись сборки приложения
- `apps/src-tauri/src/settings_commands.rs`: Команда моста настройки рабочего стола
- `apps/src-tauri/src/service_runtime.rs`: Жизненный цикл встроенных услуг настольных компьютеров
- `apps/src-tauri/src/rpc_client.rs`: Рабочий стол RPC Инфраструктура вызовов

### 3.3 сервисный шлюз и запись протокола

- `crates/service/src/lib.rs`: Главный вход для обслуживания и сборка во время выполнения
- `crates/service/src/http/`: запись маршрутизации HTTP
- `crates/service/src/rpc_dispatch/`: RPC Вход для раздачи
- `crates/service/src/gateway/mod.rs`: запись агрегации шлюза
- `crates/service/src/gateway/observability/http_bridge.rs`: отслеживание запросов, соединение протоколов, запись журналов.
- `crates/service/src/gateway/protocol_adapter/request_mapping.rs`: Сопоставление входных данных OpenAI/Codex
- `crates/service/src/gateway/protocol_adapter/response_conversion.rs`: запись об общем количестве конверсий результатов, не связанных с потоковой передачей.
- `crates/service/src/gateway/protocol_adapter/response_conversion/sse_conversion.rs`: Потоковая запись SSE о преобразовании
- `crates/service/src/gateway/protocol_adapter/response_conversion/openai_chat.rs`: OpenAI Адаптация результатов чата
- `crates/service/src/gateway/protocol_adapter/response_conversion/tool_mapping.rs`: сокращение и восстановление имени инструмента

### 3.4 Настройка и запуск записи конфигурации

- `crates/service/src/app_settings/`: настройка постоянства, охвата переменных среды, синхронизации времени выполнения.
- `crates/service/src/web_access.rs`: Web Пароль доступа и токен сеанса

## 4. Текущие отношения

### 4.1 Режим рабочего стола

Режим рабочего стола состоит из следующих частей:

- `apps/src/`: Интерфейсный интерфейс
- `apps/src-tauri/`: Оболочка рабочего стола
- `crates/service/`: местная служба

Как запустить:

1. Пользователь запускает настольное приложение.
2. Tauri Оболочка отвечает за поведение рабочего стола, такое как окна, лотки, обновления, отдельные экземпляры и настройка мостов.
3. Рабочий стол связывается с `codexmanager-service` через RPC или локальный адрес.
4. Интерфейсный интерфейс отображает такие страницы, как учетная запись, использование, журнал запросов и настройки.

### 4.2 Режим Service

Шаблон Service состоит из следующих двоичных файлов:

- `codexmanager-service`
- `codexmanager-web`
- `codexmanager-start`

Обязанности:

- `codexmanager-service`: Основной сервисный процесс, обеспечивающий управление учетными записями, переадресацию шлюза, журналы запросов, постоянство настроек и интерфейсы RPC/HTTP.
- `codexmanager-web`: Web Сервисная оболочка пользовательского интерфейса, которая может напрямую предоставлять интерфейсные страницы и прокси-сервер локальным службам.
- `codexmanager-start`: Средство запуска одним щелчком мыши для публикации пакетов, отвечающее за одновременный запуск сервиса и Интернета.

## 5. Обязанности модуля

### 5.1 `apps/src/`

Основная ответственность за:

- Рендеринг страницы
- взаимодействие с пользователем
- Управление статусами
- Вызов локального API/команды Tauri
- Интерфейсная логика страницы настроек и страницы учетной записи

### 5.2 `apps/src-tauri/`

Основная ответственность за:

- Tauri Запуск приложения
- Управление одним экземпляром
- События в системном трее и окнах
- Обновления рабочего стола и поведение установщика
- Объедините интерфейсные операции со службой/локальной средой выполнения.

### 5.3 `crates/core/`

Основная ответственность за:

- SQLite Миграция
- Базовые возможности хранилища
- Основная базовая логика, такая как аутентификация/использование.
- Возможности доступа к данным, которые могут повторно использоваться службами.

### 5.4 `crates/service/`

Основная ответственность за:

- HTTP / RPC Портал
- Учетная запись, использование, управление ключами API
- Возможности локального шлюза
- Адаптация протокола и восходящая пересылка
- Запросить ведение журнала и настроить постоянство
- Синхронизация конфигурации времени выполнения

Ключевые подкаталоги:

- `src/gateway/`: Шлюз, адаптация протокола, потоковое и непотоковое преобразование
- `src/http/`: запись маршрутизации HTTP
- `src/rpc_dispatch/`: RPC Распространение
- `src/account/`, `src/apikey/`, `src/requestlog/`, `src/usage/`: Логика предметной области

### 5.5 `crates/web/`

Основная ответственность за:

- Предоставьте статические ресурсы пользовательского интерфейса Web.
- Монтирование или прокси-сервер для службы
- При желании вставьте `apps/dist` в двоичный файл, чтобы сформировать однофайловый дистрибутив.

### 5.6 `crates/start/`

Основная ответственность за:

- Предоставьте более прямую запись запуска в пакете выпуска Service.
- Координировать жизненный цикл сервиса и сети

## 6. Данные и конфигурация

### 6.1 База данных

В текущем проекте используется SQLite.
Миграция базы данных находится по адресу:

- `crates/core/migrations/`

База данных не только хранит учетные записи, но и предполагает:

- API-ключ
- Запросить журнал
- статистика токенов
- настройки приложения

### 6.2 Запуск конфигурации

К основным источникам конфигурации относятся:

- Переменные среды `CODEXMANAGER_*`
- `.env` / `codexmanager.env` в каталоге запуска приложения
- `app_settings` Таблица устойчивости
- Страница настроек рабочего стола

Текущее соглашение:

- Конфигурации, которые должны вступить в силу перед запуском, сохраняются на уровне переменных среды.
- Настраиваемые конфигурации во время выполнения сначала управляются через страницу настроек + `app_settings`.
- Изменения настроек не должны без границ разбрасываться по рабочим столам, интерфейсам и службам.

## 7. Запросить обзор ссылки

Типичные ссылки на запросы следующие:

1. Клиент или пользовательский интерфейс инициирует запрос.
2. Запросы поступают на уровень HTTP/RPC `crates/service`.
3. Модуль шлюза определяет стратегию пересылки, номер счета, стратегию заголовка, восходящий прокси-сервер и т. д.
4. 协议适配层负责处理：
   - `/v1/chat/completions`
   - `/v1/responses`
   - Потоковая передача SSE
   - Непотоковый JSON
   - `tool_calls` / инструменты сопоставления и агрегирования
5. Результаты записываются обратно в журнал запросов и статистику, а затем возвращаются вызывающей стороне.

## 9. Текущие структурные риски

Текущий склад должен сосредоточиться на следующих вопросах:

1. `apps/src-tauri/src/lib.rs` Он все еще толстый, и сборку оболочки рабочего стола и реализацию команд все еще необходимо разобрать.
2. `crates/service/src/lib.rs` Конфигурация, синхронизация времени выполнения и границы побочных эффектов недостаточно ясны.
3. `crates/service/src/gateway/protocol_adapter/response_conversion.rs` Существует множество совместимых ветвей, и риск регресса высок.
4. `.github/workflows/release-all.yml` Тем не менее, длинная многоплатформенная логика требует ограничений постоянства.

## 10. Предлагаемые изменения

Чтобы уменьшить структурное загрязнение, новые требования должны быть ориентированы в соответствии со следующими принципами:

- Новые страницы или взаимодействие с интерфейсом: приоритет попадает в `apps/src/views/`, `apps/src/services/`, `apps/src/ui/`.
- Новые возможности рабочего стола: отдавайте приоритет автономным модулям, попадающим в `apps/src-tauri/src/`, вместо того, чтобы продолжать запихивать их все в `lib.rs`.
- Новый элемент настройки: сначала определите, принадлежит ли он переменным среды, постоянной конфигурации или состоянию времени выполнения.
- Совместимость с новыми протоколами: приоритет должен быть отдан подмодулю адаптера шлюза/протокола, и не продолжать складывать условные ветки не по порядку.
- Новая логика выпуска: отдавайте приоритет рисованию сценариев или повторному использованию шагов и не повторяйте изменения три раза на трех платформах.
