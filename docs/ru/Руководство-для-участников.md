# ВКЛАД

Этот документ используется для ограничения метода ежедневной совместной работы, предусмотренного CodexManager, с целью позволить новым сотрудникам завершить разработку, проверку, отправку и выпуск с минимально возможной устной передачей.

## 1. Позиционирование проекта

CodexManager Ни одного фронтенд-проекта, ни одного сервисного проекта Rust.
На текущем складе также имеются:

- Рабочий стол: `apps/` + `apps/src-tauri/`
- Местные услуги: `crates/service`
- Web Оболочка: `crates/web`
- Service Пусковая установка: `crates/start`
- База данных и хранения: `crates/core`
- Сценарий сборки/выпуска: `scripts/`.
- GitHub Ссылка на публикацию действий: `.github/workflows/`

Поэтому перед отправкой вы должны сначала определить, к какой границе относятся ваши изменения, и избегать объединения нескольких обязанностей непосредственно в один и тот же файл.

Запись в руководящем документе:

- `README.md`: Знакомство с проектом и быстрый старт.
- `Архитектура.md`: Структурные границы и операционные отношения
- `Тестирование.md`: Базовый план проверки на уровне репозитория.
- `Безопасность.md`: Вопросы безопасности и правила обращения с конфиденциальной информацией.
- `docs/README.md`: Каталог руководящих документов и правила подачи

## 2. Среда разработки

### 2.1 Основные инструменты

-Node.js 20
- пнпм 9
- Устойчивость к ржавчине
- Windows Для локальной упаковки требуется PowerShell 7+.
- Tauri Упаковка должна соответствовать зависимостям платформы.

### 2.2 Установка зависимостей

``` баш
pnpm -C установка приложений
грузовой тест --workspace
```

### 2.3 Часто используемые локальные команды

Интерфейс:

``` баш
pnpm -C приложения запускают разработку
pnpm -C приложения запускают тест
pnpm -C приложения запускают тест: ui
pnpm -C приложения запускают сборку
pnpm -C проверка запуска приложений
```

Ржавчина:

``` баш
грузовой тест --workspace
грузовая сборка -p codexmanager-service --release
Cargo build -p codexmanager-web --release
грузовая сборка -p codexmanager-start --release
```

Настольная упаковка:

```powershell
pwsh -NoLogo -NoProfile -File scripts/rebuild.ps1 -Bundle nsis -CleanDist -Portable
```

## 3. Соблюдайте границы

### 3.1 Изменение файлов в соответствии с обязанностями

Отдайте приоритет следующим границам:

- Главная страница, взаимодействие, статус: `apps/src/`
- Оболочка рабочего стола, лоток, окно, команда Tauri: `apps/src-tauri/src/`
- HTTP на стороне сервера / RPC / Шлюз / Адаптация протокола: `crates/service/src/`
- Миграция базы данных, инфраструктура хранения: `crates/core/`.
- Сценарии выпуска и сборки: `scripts/`, `.github/workflows/`.

### 3.2 Текущие файлы высокого риска

Следующие файлы явно слишком велики, поэтому вам следует воздержаться от добавления общей логики управления при их изменении:

- `apps/src/main.js`
- `apps/src-tauri/src/lib.rs`
- `crates/service/src/lib.rs`
- `crates/service/src/gateway/protocol_adapter/response_conversion.rs`
- `.github/workflows/release-all.yml`

### 3.3 Порог предупреждения о большом файле

При достижении следующего порога логику не следует продолжать добавлять по умолчанию, но сначала следует оценить разделение:

- JavaScript/TypeScript: оповещения начинаются на строках, превышающих `500`, а строки, превышающие `800`, должны объяснять, почему их нельзя разбивать.
- Rust: предупреждение появится для строк, превышающих `400`. Строки, превышающие `700`, должны объяснять, почему их нельзя сносить.
- Рабочий процесс/YAML: оповещения начнутся для строк, превышающих `250`. Строки, превышающие `400`, должны объяснять, почему их нельзя разбивать.
- Документация по уценке: оповещения начнутся, когда строка превысит `300`, и приоритет будет отдан вложенному документу `docs/`.

Описание:

- «Начать предупреждение» означает, что вам следует заранее определить, следует ли продолжать распределять обязанности перед отправкой.
- «Необходимо объяснить, почему не разделить» означает, что причина должна быть четко указана в инструкциях по подаче или описании PR.
- Эти пороговые значения являются долгосрочными ограничениями по обслуживанию, а не индикаторами разовой очистки.

### 3.4 Запрещенные предметы

— Не продолжайте накапливать настройки, привязки событий или ветки протоколов у главного входа.
- Не рассматривайте README как журнал изменений для долгосрочного обслуживания.
- Не меняйте сценарии, рабочие процессы или номера версий без проверки.
– Не откатывайте пользовательские изменения, которые вы не создавали.
- Не копируйте и не расширяйте встроенный скрипт в рабочем процессе выпуска повторно, сначала используйте его повторно `scripts/release/`.

## 4. Проверьте перед отправкой

### 4.1 Минимальный контрольный список

Выполните как минимум следующее содержимое в зависимости от объема изменений:

Изменения во фронтенде:

``` баш
pnpm -C приложения запускают тест
pnpm -C приложения запускают сборку
pnpm -C приложения запускают тест: ui
```

Изменения на стороне Rust/сервера:

``` баш
грузовой тест --workspace
```

Изменения в ссылке на рабочий стол/упаковку:

```powershell
pwsh -NoLogo -NoProfile -Файл скриптов/rebuild.ps1 -DryRun
```

### 4.2 Изменения, связанные с адаптацией протокола

Если следующие пути изменены, необходимо выполнить минимальную регрессионную проверку:

- `crates/service/src/gateway/`
- `crates/service/src/http/`
- `crates/service/src/lib.rs`

Минимальное покрытие:

- `/v1/chat/completions`
- `/v1/responses`
- Возврат потоковой передачи
- непотоковый возврат
- `tool_calls`/путь, связанный с инструментами

### 4.3 Изменения, связанные с настройками элементов

Если вы добавляете поля страницы настроек, переменные среды или конфигурацию постоянства, вам также необходимо подтвердить:

- Значение по умолчанию понятно?
- Нужно ли писать `app_settings`
- Влияет ли это на поведение десктопных/сервисных/веб-терминалов
- Необходимо ли обновить README или специальную документацию.

## 5. Отправьте информацию и PR-договор

### 5.1 Отправьте информацию

На нынешнем складе в основном используются инструкции по отправке на китайском языке, требующие:

- Представление решает только один тип проблем.
- Заголовок прямо описывает результаты и не пишет пустые слова
- Не вставляйте несколько несвязанных изменений в один и тот же коммит.

### 5.2 ПР описывает минимальные требования

PR должен, по крайней мере, четко заявить:

- Какие файлы были изменены?
- Какую проблему решить
- Какие платформы или интерфейсы затронуты
- Какие проверки проводились?
- Есть ли риск непокрытия?

## 6. Предварительная проверка

Перед каждым выпуском вы должны подтвердить:

1. `Журнал-изменений.md` обновлен.
2. `README.md` соответствует текущей версии `README.en.md`.
3. В соответствии с версиями `Cargo.toml`, `apps/src-tauri/Cargo.toml` и `apps/src-tauri/tauri.conf.json`.
4. Входное описание рабочего процесса выпуска, описание параметров сценария и фактический рабочий процесс должны быть согласованными.
5. Пути совместимости с высоким риском должны пройти хотя бы один этап локальной проверки.
6. Если логика наименования продукта или типа выпуска изменена, необходимо проверить `prerelease` и поведение тега.

## 7. Правила ведения документов

Договор на долгосрочное техническое обслуживание заключается в следующем:

- `README.md` / `README.en.md` Отвечает за представление проекта, быстрый старт и инструкции по входу.
- `Журнал-изменений.md` Отвечает за историю версий.
- `Архитектура.md` Отвечает за структурные границы и оперативные взаимоотношения.
- `Руководство-для-участников.md` Отвечает за правила сотрудничества и проверку перед отправкой.

Перестаньте складывать историю версий, описание архитектуры и сведения о выпуске обратно в README.

## 8. Как справиться с серьезными изменениями

Если выполняется любое из следующих условий, рекомендуется разделить задачу перед ее отправкой:

- Включает в себя одновременно три границы: интерфейс, рабочий стол и сервер.
- Изменение адаптации протокола, установка постоянства и публикация ссылок одновременно.
- Необходимо переименовать продукт, изменить рабочий процесс и скорректировать стратегию версий.
- Необходимо разделить большие файлы с высоким уровнем риска.

Предлагаемый порядок:

1. Сначала дополните сценарий тестирования или проверки
2. Снова провести реконструкцию или структурную перестройку.
3. Окончательная документация и примечания к версии
