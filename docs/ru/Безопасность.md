# БЕЗОПАСНОСТЬ

## Диапазон поддержки

CodexManager Мы по-прежнему быстро совершенствуемся, но постараемся сделать все возможное, чтобы сообщения о проблемах безопасности оставались в разумных пределах.

К приоритетным вопросам относятся:

- Токен учетной записи, токен RPC, риск утечки ключа платформы.
- Web Обход пароля доступа
- Несанкционированный доступ к локальному шлюзу
- Чувствительная утечка журналов
- Проблемы с удаленной эксплуатацией высокого риска, затрагивающие настольные компьютеры, службы и веб-терминалы.

## Метод отчета

Пожалуйста, не раскрывайте напрямую воспроизводимые конфиденциальные сведения об уязвимостях, действительные токены или данные учетной записи в публичных выпусках.

Рекомендуется связаться с сопровождающим лично через:

- контакты владельца репозитория на GitHub
- Установлены частные каналы связи.

Если на данный момент связаться можно только через раздел «Проблемы», опишите только масштаб воздействия и не указывайте:

- действительный токен доступа
-обновить токен
-идентификатор токена
- токен RPC
- Ключ API в виде обычного текста
- Скрипты атак, которые можно использовать повторно.

## Что мы хотим получить

Пожалуйста, постарайтесь предоставить:

1. Сфера влияния
2. Этапы воспроизведения
3. Условия срабатывания
4. Ожидаемые результаты и фактические результаты
5. Нужен ли вход/нужны ли условия локальной сети?
6. Фрагменты логов или скриншоты (обратите внимание на десенсибилизацию)

## Соглашения об обработке конфиденциальной информации

Прежде чем отправлять журналы, снимки экрана и конфигурации, уменьшите чувствительность следующего контента:

- `Authorization` Глава
- `Cookie`
- `Set-Cookie`
- `access_token`
- `refresh_token`
- `id_token`
- `CODEXMANAGER_RPC_TOKEN`
- Любая учетная запись агента или ключ платформы, допускающая повторное использование.

## Ограничения безопасности на складе

- Не отправляйте на склад настоящие токены, пароли, файлы cookie и ключи API.
- Не публикуйте конфиденциальные значения, которые можно использовать непосредственно в README, Issue, PR или документах.
- При добавлении новых журналов приоритет отдается записи структурированного контекста, а конфиденциальные текстовые данные не выводятся напрямую.
- При добавлении элементов настройки различайте «конфигурации, подходящие для сохранения» и «конфиденциальные значения, которые могут быть предоставлены только в переменных среды».

## Текущие известные границы

- Проект по умолчанию использует сценарии локального развертывания и самостоятельного использования и не обещает полностью автоматического усиления безопасности при доступе к общедоступной сети.
- Если включен мониторинг `0.0.0.0` или Web/услуга доступна в локальной сети или общедоступной сети, разработчику необходимо принять на себя риск заражения сети и сотрудничать со:
  - Надежный пароль
  - Контроль доступа к обратному прокси-серверу
  - Ограничения границ сети

## Принципы реагирования

- Подтвержденные проблемы будут воспроизведены в максимально возможной степени и оценен уровень воздействия.
- После завершения ремонта он будет выпущен в соответствующей версии, и конфиденциальные сведения не будут заранее раскрыты по общедоступным каналам.
- Если проблема связана не с самим складом, а с неправильной настройкой развертывания, также будет дано описание границ.
