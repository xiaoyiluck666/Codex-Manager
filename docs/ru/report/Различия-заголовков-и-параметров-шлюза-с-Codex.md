# Текущий шлюз и таблица сравнения передачи параметров Codex

Примечание. В текущей рабочей области собраны исходящие прямые контрактные разницы Codex. Следующее реорганизуется в соответствии с «Как переместить параметры / Как в конечном итоге выйти из станции / Выровнено ли она по восходящему течению».

## 1. Ссылка для передачи параметров

1. Входящий HTTP-запрос сначала поступает в `crates/service/src/gateway/request/incoming_headers.rs`, где делается только снимок заголовка и запрос не перезаписывается напрямую.
2. Привязка сеансов рассчитывается равномерно по `crates/service/src/gateway/request/session_affinity.rs`, что дает `incoming_session_id`, `incoming_client_request_id` и `fallback_session_id`.
3. Тело запроса вступает в процесс перезаписи в `crates/service/src/gateway/request/request_rewrite.rs`, а затем поле совместимости ответов обрабатывается в `request_rewrite_responses.rs`.
4. Окончательный исходящий заголовок формируется с помощью `crates/service/src/gateway/upstream/headers/codex_headers.rs`.
5. Перед фактической отправкой в ​​восходящий поток заголовки + тело передаются запросу в соответствии с `crates/service/src/gateway/upstream/attempt_flow/transport.rs`.

| `x-codex-turn-metadata` | Прозрачное входящее значение | Также будет осуществляться исходящий исходящий HTTP-трафик; Сторона WS также будет упакована в `client_metadata` | Выровнено | Текущий шлюз дополнительно не создает ту же модель WS `client_metadata` |
| `x-codex-turn-state` | Сохраняется при прямом подключении, удаляется при `strip_session_affinity` | восходящий поток также зависит от этого состояния сеанса | Выровнено | Отправляется только в том случае, если близость не удалена |
| `OpenAI-Beta` | Отправлен запрос на потоковую передачу `responses_websockets=2026-02-06` | Восходящий путь WebSocket имеет то же значение | Выровнено | Запрос на потоковую передачу не отправляется |
| `x-responsesapi-include-timing-metrics` | Потоковая передача и отправка при включении `true` | Upstream имеет ту же семантику | Выровнено | Не отправляется, если нет потоковой передачи |

## 4. Заключение

1. Заголовки исходящих запросов Codex и параметры запроса текущего шлюза сохранили ту же форму, что и восходящий поток.
2. Сейчас остаются в основном различия на уровне реализации, такие как источник версии `User-Agent` и упаковка `client_metadata` `x-codex-turn-metadata` в сценарии WS.
3. `stream_passthrough`, внутренняя идентификация учетной записи, откат привязки сеанса и т. д. — все это относится к внутренней логике уровня адаптации и больше не учитывается в Codex прямого запроса на разницу в договоре.
