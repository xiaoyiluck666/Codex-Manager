# Часто задаваемые вопросы и правила попадания в аккаунт

## Часто задаваемые вопросы
- Ошибка обратного вызова при авторизации: установите приоритет проверки того, занят ли `CODEXMANAGER_LOGIN_ADDR`, или используйте разрешение обратного вызова вручную в пользовательском интерфейсе.
- Список моделей или запрос перехватываются вызовом: установите приоритет проверки выхода прокси, различий в заголовках запроса и состояния учетной записи и больше не устраняйте неполадки с помощью старых резервных или восходящих файлов cookie.
- Все еще заблокировано Cloudflare/WAF: не устраняйте неполадки по старому пути совместимости, а в будущем действуйте только в соответствии с семантикой `Codex-First`.
- Часто появляется сообщение «Не удалось обновить некоторые данные, были отображены доступные данные»: сценарий автоматического обновления изменен на запись только журналов; при ручном обновлении будут отображаться неудавшиеся элементы и образцы ошибок. Установите приоритет проверки интервала/переключателя «Фоновая задача» на странице настроек и имени невыполненной задачи в журнале службы.
- Независимый запуск службы/Web: если каталог недоступен для записи (например, каталог установки), установите для `CODEXMANAGER_DB_PATH` доступный для записи путь.
- macOS Запросы в среде прокси `502/503`: установите приоритет для подтверждения того, что системный прокси не принимает на себя локальный запрос шлейфа (`localhost/127.0.0.1`, перейдите к `DIRECT`), и убедитесь, что адрес использует строчные буквы `localhost:<port>`.

## Инструкции по миграции

### Codex-Первое направление

Текущий склад определил направление последующих действий как `Codex-First`:

- Больше не используйте старое совместимое поведение в качестве основного пути.
- К одному сеансу может быть привязана только одна активная учетная запись.
- Переключение учетных записей вручную приведет к обрезанию тем, а автоматическое переключение учетных записей также приведет к обрезанию тем.

### Разница между текущим поведением и целевым поведением

Текущее поведение:

- `balanced` По-прежнему строгий опрос согласно `Key + 模型`
- Семантика привязки сеансов и учетных записей по-прежнему мигрирует с «опроса по запросу» на «привязку по сеансу».

Целевое поведение:

- Связанные сеансы сначала затрагивают связанную учетную запись и больше не будут участвовать в каждом раунде обычного опроса.
- Автоматическое переключение учетных записей синхронно переключает генерацию восходящего потока.
- Старый переключатель совместимости полностью вышел из основного пути и больше не используется в качестве рекомендуемой конфигурации.

## Правила попадания в аккаунт
- В режиме `ordered` (приоритет порядка) шлюз строит кандидатов в порядке возрастания учетной записи `sort` и пробует их последовательно, например `0 -> 1 -> 2 -> 3`.
- Это означает «пробовать по порядку», а не «всегда нажимать на цифру 0»; если предыдущая учетная запись недоступна или вышла из строя, она автоматически переключится на следующую.

### Распространенные причины, по которым учетные записи предварительного заказа не активируются
- Статус аккаунта не `active`
- В аккаунте отсутствует токен
- Определение использования недоступно, например, главное окно исчерпано и поле использования отсутствует
- Учетная запись находится в режиме восстановления, или одновременное срабатывание мягкого ограничения срабатывает.

### `balanced` Режим
- Режим `balanced` (сбалансированный циклический алгоритм) по умолчанию будет чередоваться между всеми доступными учетными записями строго по размеру `Key + 模型`, без гарантии начала с минимального `sort`.
- Обмены здоровья будут добавлены в заголовок сбалансированного опроса только в том случае, если вы явно увеличите `CODEXMANAGER_ROUTE_HEALTH_P2C_BALANCED_WINDOW`.

## Журнал устранения неполадок
Вы можете просмотреть тот же каталог базы данных `gateway-trace.log`:
- `CANDIDATE_POOL`: Приказ-кандидат по этому запросу
- `CANDIDATE_START` / `CANDIDATE_SKIP`: фактическая попытка и причина пропуска.
- `REQUEST_FINAL`: Последний хит-аккаунт
